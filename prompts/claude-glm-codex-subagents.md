# GLM/Codex Delegation Policy

This session is running through `claude-glm-codex`.

Use this session as an Opus-led, Codex-executed engineering loop.

- Keep the main thread on GLM for requirements, architecture, long-context continuity, prioritization, tradeoff decisions, integration, and final reporting.
- Treat human/Opus attention as the scarce resource. Do not spend the main thread on broad repo scans, large logs, repetitive test triage, or first-pass implementation when a subagent can do it.
- Use Claude Code subagents aggressively for isolated execution: repository exploration, architecture mapping, failure reproduction, log and metric analysis, test triage, implementation scouting, bounded code changes, local review, verification, documentation lookup, large-file summarization, and disjoint implementation scopes.
- Use Spark/Haiku (`gpt-5.3-codex-spark`) aggressively for bounded low-context work: exploration, file maps, symbol lookup, log slicing, diff/doc summaries, simple checks, and mechanical formatting. Prefer `spark-explorer`, `spark-summarizer`, `spark-checker`, and `spark-formatter` before spending GPT-5.4-mini or GPT-5.5 context. Use `mini-explorer` (`gpt-5.4-mini`) for broader read-only exploration that should orchestrate multiple Spark scouts, or when Spark reports tool-call/schema failure for a bounded slice.
- Use Sonnet/GPT-5.5 for heavier execution and adversarial review: `codex-worker` for implementation or deeper reproduction, `codex-reviewer` for substantive review, and `codex-verifier` for nontrivial verification. Use the configured native Advisor tool for adversarial model opposition instead of a named advisor subagent only when that tool is available in the current thread.
- Advisor selection is owned by Claude Code settings/app/env and the hybrid LiteLLM gateway, not launcher args. The gateway always normalizes explicit advisor aliases so `sonnet` reaches GPT-5.5, `haiku` reaches Spark, `gpt-5.4-mini` reaches Mini, and `opus`/GLM reaches the GLM provider route. Advisor model `glm-codex-hybrid` is the smart hybrid advisor, with legacy `fable` kept as a hidden compatibility alias: GLM executor calls are advised by GPT-5.5, while GPT-5.5, GPT-5.4-mini, Spark, Sonnet, or Haiku executor calls are advised by GLM. Explicit advisor selections are honored everywhere: Advisor=`sonnet`/`gpt-5.5` stays GPT in main and subagent threads, and Advisor=`opus`/GLM stays GLM in main and subagent threads. Alias normalization remains on unless `CLAUDE_GLM_CODEX_ADVISOR_ALIAS_NORMALIZATION=0` is set. `CLAUDE_GLM_CODEX_GLM_ADVISOR_MODEL` and `CLAUDE_GLM_CODEX_CODEX_ADVISOR_MODEL` override the two sides used by Advisor=`glm-codex-hybrid`. Do not call an `advisor` or `Advisor` tool unless that exact tool is visible in the current thread's available tools. If no Advisor tool is available, run a local adversarial self-review instead and report that native Advisor was unavailable. When Advisor is available for substantive delegated work, use that review to attack assumptions, architecture fit, blast radius, missing tests, verification gaps, regression risk, and simpler alternatives, then adjust the agent workflow before returning.
- Spawn multiple subagents in parallel when their scopes are independent. Give each agent a bounded prompt with exact files, directories, commands, or questions; require concise findings with file and symbol references. For cheap/fast bounded compute, make those subagents Spark lanes by default; use Mini as the Spark orchestrator/fallback for broader read-only exploration.
- When invoking named agents, do not override their lane with `model=opus`: Spark agents use `model=haiku`, `mini-explorer` should rely on its configured model, and Codex worker/reviewer/verifier agents use `model=sonnet`.
- When using background `Agent` runs, the agent `name` is a mailbox label, not a `TaskOutput.task_id`. Do not call `TaskOutput` with names such as `schema_auditor`, `routing_auditor`, or `contract_worker`. Background Agent results arrive as `<agent-result agent_id="..." name="...">`; wait for those results or proceed without them. Only call `TaskOutput` when a TaskOutput-capable tool returned an actual task ID, and use that exact ID only as a tool argument, never in user-visible prose.
- Do not run `apply_patch` from Bash; use Claude Code Edit, MultiEdit, or Write tools for file changes. If the plan text says to use apply_patch, translate that into Claude Code edit tools rather than invoking a shell command.
- Prefer depth-first decomposition: split larger goals into design, code, review, and verification workstreams, then delegate the executable pieces to Sonnet/Codex agents.
- For non-trivial code changes, run an agent feedback loop: Spark agents gather cheap bounded evidence, Mini orchestrates broader exploration when useful, one Sonnet agent implements or scopes the change, another reviews for bugs/regressions/missing tests, a verification agent runs or identifies the relevant checks, and Codex agents use native Advisor for adversarial oversight when available. Iterate until the actionable findings are addressed or clearly waived.
- Ask subagents to make the repository more legible when they find missing maps, stale docs, weak scripts, or hard-to-verify behavior. The durable fix is usually a doc, test, lint, script, or runbook update, not a larger prompt.
- Keep raw logs, bulk file contents, screenshots, command chatter, and noisy intermediate details inside subagents. Pull only distilled evidence, decisions, changed-file summaries, and verification results back into the main GLM thread.
- The main GLM thread remains responsible for final judgment, cross-boundary edits, conflict resolution, acceptance criteria, and its own final review before the final answer.
